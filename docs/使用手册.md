# 使用手册

## 推荐工作流

1. 在仓库的 `projects/` 下新建项目文件夹，把收到的原始素材直接放入项目根目录。
2. 如有已确认的品牌事实，在项目根目录添加 `design-factory.json`。
3. 执行 `npm run analyze -- --project "项目名称"`；需要实时案例候选时添加 `--online`。
4. 系统自动创建 `input/` 和 `outputs/`，安全整理根目录素材后开始分析。
5. 人工复核 Brand Lock 中的待确认项，并更新配置后重新生成。
6. 检查 `Knowledge-Candidate.md` 与 `Knowledge-Analysis.md`，按清单人工决定是否维护 Approved Rule。
7. 按图片规划制作，最后把 Chat 生图任务包交给执行者。

当 `projects/` 中只有一个项目时可省略 `--project`。旧版直接路径模式仍然兼容，例如 `node bin/design-factory.js analyze ./my-brand`。详细规则见[Projects 自动初始化](./项目自动初始化.md)。

## Knowledge Analysis

项目配置中的 `knowledgeCandidates` 是 Knowledge Engine 的结构化输入。每条候选支持：

- `id`、`title`、`category`、`content`、`reason`
- `targetRule`：明确建议更新的 Rule ID
- `verifiedProjects` 或 `evidenceCount`：跨项目证据数量
- `projectOnly`、`projectOnlyReason`：明确标记项目经验
- `priority`：可选的 P0–P3 人工优先级

分类只允许 Packaging、Brand、VI、Poster、Portfolio。未提供候选时，系统仍会把品牌色、盒型、视觉偏好和本次缺图计划记录为 Project Only，并明确没有发现新的通用设计规律。

默认读取系统 `knowledge/approved/`。可以只读指定其他目录：

```bash
node bin/design-factory.js analyze ./my-brand --knowledge-dir /path/to/approved
```

Approved Rule 支持 JSON，或包含 `id`、`title`、`category` Frontmatter 的 Markdown。分析过程不会修改该目录。

## 支持范围

| 格式 | 能力 |
|---|---|
| ZIP | 读取中央目录、文件名与数量，不解压到磁盘 |
| PDF | 页数、基础元数据与可见文本线索；扫描 PDF 仅能统计页数 |
| PPTX/POTX/PPSX | 页数、幻灯片文本、主题色与主题字体 |
| 旧版 PPT | 纳入清单，暂不深度解析 |
| PNG/JPEG/GIF/WebP/SVG/BMP | 格式和可用尺寸；SVG 额外提取颜色与文本 |
| MD/TXT/JSON/CSV | 文本线索，最多读取前 200KB |

## 常见问题

### 为什么没有自动确定主色？

主色是品牌事实。只有配置、PPTX 主题或 SVG 明确色值时，系统才输出候选；仅凭压缩图片文件名无法可靠判定时会要求人工确认。

### 联网失败是否阻塞？

不阻塞。系统记录失败原因并回退到内置、按行业分类的策展案例库。生产项目应人工复核联网候选的质量与可访问性。

### 可重复运行吗？

可以。默认 `outputs/` 在扫描时会被忽略，避免上一次报告被当作新素材。相同输入和配置在离线模式下产生一致的业务数据，仅生成时间不同。
