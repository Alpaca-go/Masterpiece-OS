## 修复：启动卡死 + Markdown code fence 解析错误

### 启动卡死修复（上一轮）
- **根因**：`settings-store.ts` 默认数据目录用 `app.getPath('documents')`（Windows「文档」文件夹）。
  当该路径是网络重定向/OneDrive 离线/企业漫游配置文件时，`getPath` 同步阻塞或 `readdir` 超时，
  导致所有依赖 `getSettings()` 的 IPC 永不返回 → splash 永久卡死且无报错。
- **修复**：默认目录改为 `app.getPath('userData')`（本地应用数据，永不需要远程解析）。
- **额外防护**：渲染端 `App.tsx` 的 `refresh()` 加了 20 秒超时兜底，避免以后再静默卡死。

### Markdown code fence 解析错误（本轮）
- **现象**：用户运行客户端的「参考风格重构」流程，在步骤 `SELECTING_REFERENCE_MASTER_SET` 报错：
  「模型输出使用了 Markdown 代码块包装，而协议要求 JSON。」
- **根因**：`pipeline-service.ts:127-138` 的 `parseModelStructuredResponse()` 在检测到响应以 ```` 开头时直接抛出 `markdown_wrapper` 阻塞性错误。但模型经常忽略 prompt 中的「只返回 JSON」指令而用 ```json ... ``` 包装。
- **对比**：引擎层 `src/v5/shared/analysis/response-parser.js` 的 `extractJsonCandidate()` 正确地先 `.replace(/^```(?:json)?\s*/i, '')` 剥离再解析。
- **修复**：删除 `pipeline-service.ts` 中的「检测到 code fence 就报错」逻辑，改为先 strip 再交给 `parseStructuredResponse` 解析，与引擎层行为一致。
- **重新打包**：产物已更新为含此修复的便携版 exe。
